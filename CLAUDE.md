# CLAUDE.md — 니트커피(Knit Coffee) 도매 주문 시스템

> 이 문서는 Claude Code가 프로젝트를 이해하기 위해 매 세션 자동으로 읽는 컨텍스트 파일입니다.
> 기존 개발 과정에서 확정된 규칙과 정책을 담고 있으니, 코드 수정 시 아래 규칙을 반드시 준수하세요.

---

## 1. 프로젝트 개요

- **이름:** 니트커피(Knit Coffee) 도매 주문 시스템 (웹 앱)
- **운영자:** 이강민 (니트커피 대표, 개인사업자)
- **목적:** 도매 거래처의 원두 주문 접수 + 샘플 신청 + 경영/재무 관리 + OEM 공장(클라리멘토) 채무 관리를 하나의 웹 앱으로 통합
- **로스터리:** 원두는 '클라리멘토'를 통해 OEM 생산·납품

---

## 2. 기술 스택

- **백엔드:** Express.js (Node.js)
- **프론트엔드:** React
- **디자인:** Tailwind CSS
- **데이터베이스:** SQLite (파일명: `data.db`, 프로젝트 루트에 위치)
- **배포:** GitHub → Railway 자동 배포 (push 시 자동 반영)
- **알림:** 주문 발생 시 백엔드에서 실시간 주문 알림 이메일 직접 발송

---

## 3. 사용자 권한 구조 (중요 — 용어 주의)

과거 대화에서 용어 혼동이 있었으므로 아래 정의를 정확히 따르세요.

- **소유자(Owner):** 이강민 대표. 모든 기능 접근 가능.
- **관리자(Admin/직원, = 기존 대화의 '매니저'):** 기록·입력만 가능.
- **경영·재무 메뉴 전체(경영 대시보드·방문자 통계·매장매출·지출·가계부·고정비 항목·상품 카테고리)는 소유자 전용.** 직원은 메뉴가 보이지 않고, URL 직접 접근도 막히며, 관련 서버 API도 requireOwner로 차단됨. (매장매출·지출 입력도 직원 불가로 변경됨 — 2026-07 정책 변경)

---

## 4. 도매 주문 규칙

- **최소 주문:** 5kg 이상
- **5kg 판정 기준:** "원두 카테고리" 상품의 **주문 수량 합계**로 판단 (중량 환산 없이 개수 기준, **샘플 제외**)
- **관리자 대리 주문:** 소유자 + 관리자(직원)가 입력 가능. 입력 즉시 상태를 '처리완료'로 하여 클라리멘토 자동발주까지 연결.

---

## 5. 샘플 신청 시스템

- **가격:** 무료 (승인된 거래처에 한해 **1회만**)
- **신청 제약:** 원두 **최대 2종**, 각 **1kg씩**만 신청 가능
- **신청 페이지:** 별도의 샘플 전용 신청 페이지 존재 (일반 주문과 분리)
- **전환 인센티브:** 없음 (샘플 주문만 깔끔하게 처리)
- **주문 식별:** 관리자 주문 목록에서 "샘플" 뱃지로 표시 + 필터링 가능

### 샘플 신청 승인 흐름 (2갈래)
1. **사업자등록 O (정상 업체):** 가입 → 자동승인 → 바로 무료 샘플 1회 신청 가능
2. **사업자등록 X (미등록 업체):** 가입은 되지만 자동승인 안 됨 → 관리자에게 승인 요청 → 소유자가 직접 승인 → 이후 무료 샘플 1회 신청 가능

- **판별 기준:** 가입 시 사업자등록번호 입력란. 채워져 있으면 자동승인 / 비어 있으면 승인 대기.
- **검증:** 사업자등록번호 형식 검증 필요.

## 6. 재무 / 경영 관리

- **매출 입력:** 매장 매출은 일별 수기 입력
- **재무 카테고리 분류:** 지출·매출 입력 시 다음으로 구분
  - 니트커피 매장
  - 홀세일(도매)
  - 온라인 판매
  - 기타 지출
- **카드 수수료:** (미확정 — 이강민님과 재논의 필요. 과거 "아예 안 다루는 걸로" 방향으로 정리된 바 있음)

### OEM 공장(클라리멘토) 채무 관리 — 관리자 전용 화면 4개
- 기존 '거래처 미수금' 화면 구조를 공장 쪽으로 미러링한 형태
- 4개 화면 모두 관리자(소유자) 전용, 직접 입력·수정 방식

### 공장 매입 회계 처리 (발생주의) — 2026-07 확정
- **공장 매입(발주) = 발생 시점에 홀세일 '지출'로 인식.** 경영 대시보드의 홀세일 지출 = 해당 기간의 발주(purchases) 총액 + 수기 입력 홀세일 지출.
- **공장 '지급'(채무 상환)은 지출로 잡지 않음.** 지급은 미지급금(채무)을 갚는 것일 뿐 비용이 아님. 공장 채무 잔액 = 발주 − 지급 (채무 관리 화면 그대로 유지).
- 지출은 별도 저장이 아니라 발주 테이블을 매번 합산해 계산 → **발주를 삭제하면 지출에서 자동 제외.**
- **도매 주문을 취소하면 그 주문으로 자동 생성된 공장 발주도 함께 삭제**(수입·지출 동시 제거). 단, 이미 이카운트로 전송된 발주는 이카운트에서 수동 삭제 필요.
- 상품 카테고리는 DB에서 관리(생성·수정·삭제·순서·원두/샘플 여부). '원두'로 표시된 카테고리만 5kg 최소주문 수량에 포함, '샘플 대상'만 무료 샘플 신청 가능.

### 매장 내부 계정 / 발주 부문 분리 — 2026-07 확정 (매장 매출원가 집계)
- 매장과 도매는 **같은 사업자**(자기거래). 매장이 원두를 쓰는 건 세금계산서 대상이 아니므로 '내부 이동'으로 처리한다.
- 거래처(customers)에 `isStore`(매장 내부 계정) 플래그. 발주(purchases)에 `segment`("wholesale" | "store") 부문 태그.
- **매장 계정(isStore=1)의 주문**: ① 도매 매출에서 제외(자기거래) ② ECOUNT 판매전표(세금계산서) 전송 차단 ③ 그 주문의 클라리멘토 자동발주는 `segment="store"`로 태그.
- **재무제표/대시보드 매출원가**: 발주를 segment별로 분할 — store 발주 → 음식점업(매장) 매출원가, wholesale 발주 → 원두도매 매출원가. **매장 원가를 도매 위에 '더하는' 게 아니라 발주 자체를 부문으로 '쪼개서' 이중계상을 방지**한다. (매장 원가 = 매입원가 기준, 즉 클라리멘토 발주 원가)
- 매장 계정은 ECOUNT 거래처 자동등록도 제외(동일 사업자번호 → 거래처코드 충돌 방지).
